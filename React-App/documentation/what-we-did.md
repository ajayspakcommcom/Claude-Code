|---------|------|--------------|

| `Link activeOptions` | `__root.tsx` (exact on Home), `products/index.tsx` (includeSearch on category tabs) | `exact: true` prevents `/` matching every route; `includeSearch: true` matches only when search params also match |
| Lazy routes | `lazy-demo.tsx` + `components/HeavyPage.tsx` | `lazyRouteComponent(() => import(...))` — separate JS chunk, only downloaded on first visit |
| Scroll restoration | `main.tsx` | `scrollRestoration: true` — browser remembers scroll position on back/forward |
| `defaultPendingMs` | `main.tsx` | 500ms threshold before showing pendingComponent (avoids flash on fast responses) |
| `defaultPendingMinMs` | `main.tsx` | 300ms minimum — keeps spinner visible long enough to avoid jarring flicker |
| Route masking | `products/index.tsx` | `<Link mask={{ to: '/products' }}>` — real route renders, browser URL shows masked URL |
| Parallel loaders | `products/$productId.tsx` | `Promise.all([fetchProduct, fetchReviews])` — both fetches run simultaneously |
| Context in loaders | `main.tsx` (context: { theme }), `__root.tsx` (createRootRouteWithContext), `products/index.tsx` | typed context flows from `createRouter` → every loader's `context` argument |

#### Key decisions / gotchas learned this session
- `useNavigate` with a reducer `search: (prev) => ...` gets typed as `never` if you don't scope it with `to` — always include `to` when updating search params so the router knows which route's schema to use.
- `createRootRouteWithContext<RouterContext>()` (not `createRootRoute()`) is required for `context` in loaders to be typed — changing `__root.tsx` is the correct approach.
- `loaderDeps` is required when search params should trigger a loader re-run — without it, changing the URL search params does nothing to the loader.
- `lazyRouteComponent` build output: `HeavyPage-xxxx.js` appears as a separate chunk in Vite's build output — confirms code splitting is working.

### ✅ INTERMEDIATE — Routing — Code-based (Complete)

**Project**: `React-App-CodeRouter/` (Vite, same port as FileRouter — run separately)

#### What is code-based routing?
Instead of using a file/folder naming convention + Vite plugin, you define every route manually in TypeScript and wire the tree together yourself.

#### Run code-based routing app:
```bash
cd React-App-CodeRouter
npm run dev        # → http://localhost:5173
```

#### Key difference from file-based routing

| Task | File-based | Code-based |
|------|-----------|------------|
| Add a route | Create `src/routes/about.tsx` | `createRoute({ path: '/about' })` in `router.tsx` |
| Dynamic param | Name file `$userId.tsx` | `path: '$userId'` in `createRoute()` |
| Nested routes | Put files in a folder | `route.addChildren([childRoute])` |
| Root layout | Create `__root.tsx` | `createRootRoute({ component: Layout })` |
| Route tree | Auto-generated by Vite plugin | Assembled manually with `addChildren()` |
| No code generation | ❌ needs `routeTree.gen.ts` | ✅ pure TypeScript |
| Vite plugin | Required | Not needed |

#### Concepts demonstrated in React-App-CodeRouter

| Concept | File | Notes |
|---------|------|-------|
| `createRootRouteWithContext<T>()` | `router.tsx` | Root layout + typed context for all loaders |
| `createRoute()` | `router.tsx` | Every route defined explicitly with `getParentRoute` |
| `addChildren()` | `router.tsx` | Nesting: `usersRoute.addChildren([userDetailRoute])` |
| `createRouter()` + `context` | `router.tsx` | Assembles tree + passes `{ theme: 'light' }` to all loaders |
| `getRouteApi()` | All page files | Access typed route hooks without circular deps |
| `beforeLoad` guard | `router.tsx` (dashboardRoute) | Same API as file-based — redirect to login |
| `loader` + `staleTime` | `router.tsx` (usersRoute, productsRoute) | Fetch before render, cache result |
| `pendingComponent` | `router.tsx` (multiple routes) | Spinner while loader runs |
| `errorComponent` | `router.tsx` (userDetailRoute, productDetailRoute) | Per-route error boundary |
| `validateSearch` + `loaderDeps` | `router.tsx` (productsRoute) | Typed search params with Zod |
| Context in loaders | `router.tsx` (productsRoute loader) | `context.theme` — typed via `createRootRouteWithContext` |
| Parallel loaders | `router.tsx` (productDetailRoute) | `Promise.all([fetchProduct, fetchReviews])` |
| Route masking | `pages/Products.tsx` | `<Link mask={{ to: '/products', search: {...} }}>` |
| Lazy routes | `router.tsx` (lazyRoute) + `pages/LazyPage.tsx` | `lazyRouteComponent(() => import(...))` — separate chunk confirmed in build |
| `useSearch` | `pages/Products.tsx` | `routeApi.useSearch()` — typed category + sort |
| `useNavigate` | `pages/Contact.tsx` | All variants: to, search, params, replace |
| `useParams` (nested) | `pages/UserDetail.tsx`, `pages/ProductDetail.tsx` | `useParams({ from })` in child component |
| `Link activeOptions` | `pages/Products.tsx` | `exact + includeSearch` on category tabs |
| `scrollRestoration` + `defaultPendingMs` | `router.tsx` (createRouter) | Same config options as file-based |

#### Gotchas learned
- `getRouteApi('/path')` solves the circular dependency: page files can't import the route object from `router.tsx` (which imports the page) — `getRouteApi()` looks up by path string at runtime, fully typed.
- `createRootRouteWithContext<T>()` uses **double-call** syntax: `createRootRouteWithContext<RouterContext>()({ component: ... })` — the first call sets the type, the second call sets the options.
- `lazyRouteComponent` build output: `LazyPage-xxxx.js` is a separate chunk — confirmed in `npm run build` output.

#### 04_TanStackQuery.tsx — TanStack Query (React Query)

**Demo:** Posts list, single post lookup, parallel user fetch, pagination, add/update/delete, infinite scroll, prefetch

| Concept | Detail |
|---------|--------|
| `QueryClient` + `QueryClientProvider` | Central cache store — created once, wraps the app |
| `useQuery({ queryKey, queryFn })` | Fetch + cache. Key = cache identity. Same key = no duplicate fetch |

| Dependent queries | `enabled: !!postQ.data` — comments query waits for post query to succeed |
| Parallel queries | Multiple `useQuery` calls in one component — all fire simultaneously |
| `keepPreviousData` | Old page stays visible while next page loads — no blank flash |
| `useMutation` — add | `invalidateQueries(["posts"])` triggers list re-fetch after POST |
| `useMutation` — update | `setQueryData` patches cache directly — no extra network call |
| `useMutation` — delete | Optimistic update: remove from cache instantly, rollback via `onError` |
| `cancelQueries` | Cancels in-flight refetches before optimistic update to avoid race conditions |
| `useInfiniteQuery` | `data.pages` accumulates pages. `getNextPageParam` returns next page arg |
| `prefetchQuery` | Warms cache on hover — click renders instantly with no spinner |
| `useQueryClient()` | Access `invalidateQueries`, `setQueryData`, `getQueryData`, `prefetchQuery` in components |
| `ReactQueryDevtools` | Floating panel — inspect cache, status, staleTime, refetch live |

**When to use:** Any app fetching from REST/GraphQL APIs — the most popular server-state library in the React ecosystem.

#### 01_MemoOptimization.tsx — Memoization

**Demo:** Live render counters on every component to make re-renders visible

| Concept | Detail |
|---------|--------|
| `React.memo(Component)` | Skip re-render when props unchanged (shallow equal) |
| Custom `areEqual` | `memo(Component, (prev, next) => prev.id === next.id)` — only re-render when id changes |
| `useMemo(fn, deps)` | Cache expensive filter/sort (500-item list with 5ms artificial delay) |
| `useCallback(fn, deps)` | Stable function refs — without it, memo'd children re-render even when nothing changed |
| Common mistakes | Inline `{}` and `[]` literals create new references every render → break `React.memo` |
| Fix | `useMemo(() => ({ label: "x" }), [])` + `useCallback(() => fn(), [])` |
| When NOT to memo | Simple components, cheap computations, deps change every render — overhead costs more than it saves |

#### 03_ErrorHandling.tsx — 6 demos
| Section | Concepts |
|---------|----------|
| Error classification | `classifyError()` maps AxiosError → `{ type, message, detail, retryable }` |
| Retry + backoff | `fetchWithRetry()` — 3 attempts, 500ms/1s/2s delays, skip non-retryable errors |
| Optimistic updates | Save snapshot → update UI → API call → rollback on failure |
| Toast notifications | Non-blocking, auto-dismiss (3s), click-to-dismiss, success/error/info variants |
| Error Boundary | `getDerivedStateFromError` + `componentDidCatch` — catches render crashes |
| Global interceptor | Axios response interceptor routes ALL errors to one handler (Sentry, logging) |

**Error classification rules:**
- `!err.response` → network error (no internet, CORS, DNS) — retryable
- `err.code === "ECONNABORTED"` → timeout — retryable
- `status 401/403` → auth error — NOT retryable
- `status 404` → not found — NOT retryable
- `status >= 500` → server error — retryable

**Optimistic update pattern:** `const previous = [...items]` → update state → await API → on error: `setItems(previous)`.
