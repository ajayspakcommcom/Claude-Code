#### Key decisions / gotchas learned this session
- `useNavigate` with a reducer `search: (prev) => ...` gets typed as `never` if you don't scope it with `to` тАФ always include `to` when updating search params so the router knows which route's schema to use.
- `createRootRouteWithContext<RouterContext>()` (not `createRootRoute()`) is required for `context` in loaders to be typed тАФ changing `__root.tsx` is the correct approach.
- `loaderDeps` is required when search params should trigger a loader re-run тАФ without it, changing the URL search params does nothing to the loader.
- `lazyRouteComponent` build output: `HeavyPage-xxxx.js` appears as a separate chunk in Vite's build output тАФ confirms code splitting is working.

### тЬЕ INTERMEDIATE тАФ Routing тАФ Code-based (Complete)

**Project**: `React-App-CodeRouter/` (Vite, same port as FileRouter тАФ run separately)

#### What is code-based routing?
Instead of using a file/folder naming convention + Vite plugin, you define every route manually in TypeScript and wire the tree together yourself.

#### Run code-based routing app:
```bash
cd React-App-CodeRouter
npm run dev        # тЖТ http://localhost:5173
```

#### Key difference from file-based routing

| Task | File-based | Code-based |
|------|-----------|------------|
| Add a route | Create `src/routes/about.tsx` | `createRoute({ path: '/about' })` in `router.tsx` |
| Dynamic param | Name file `$userId.tsx` | `path: '$userId'` in `createRoute()` |
| Nested routes | Put files in a folder | `route.addChildren([childRoute])` |
| Root layout | Create `__root.tsx` | `createRootRoute({ component: Layout })` |
| Route tree | Auto-generated by Vite plugin | Assembled manually with `addChildren()` |
| No code generation | тЭМ needs `routeTree.gen.ts` | тЬЕ pure TypeScript |
| Vite plugin | Required | Not needed |

#### Concepts demonstrated in React-App-CodeRouter

| Concept | File | Notes |
|---------|------|-------|
| `createRootRouteWithContext<T>()` | `router.tsx` | Root layout + typed context for all loaders |
| `createRoute()` | `router.tsx` | Every route defined explicitly with `getParentRoute` |
| `addChildren()` | `router.tsx` | Nesting: `usersRoute.addChildren([userDetailRoute])` |
| `createRouter()` + `context` | `router.tsx` | Assembles tree + passes `{ theme: 'light' }` to all loaders |
| `getRouteApi()` | All page files | Access typed route hooks without circular deps |
| `beforeLoad` guard | `router.tsx` (dashboardRoute) | Same API as file-based тАФ redirect to login |
| `loader` + `staleTime` | `router.tsx` (usersRoute, productsRoute) | Fetch before render, cache result |
| `pendingComponent` | `router.tsx` (multiple routes) | Spinner while loader runs |
| `errorComponent` | `router.tsx` (userDetailRoute, productDetailRoute) | Per-route error boundary |
| `validateSearch` + `loaderDeps` | `router.tsx` (productsRoute) | Typed search params with Zod |
| Context in loaders | `router.tsx` (productsRoute loader) | `context.theme` тАФ typed via `createRootRouteWithContext` |
| Parallel loaders | `router.tsx` (productDetailRoute) | `Promise.all([fetchProduct, fetchReviews])` |
| Route masking | `pages/Products.tsx` | `<Link mask={{ to: '/products', search: {...} }}>` |
| Lazy routes | `router.tsx` (lazyRoute) + `pages/LazyPage.tsx` | `lazyRouteComponent(() => import(...))` тАФ separate chunk confirmed in build |
| `useSearch` | `pages/Products.tsx` | `routeApi.useSearch()` тАФ typed category + sort |
| `useNavigate` | `pages/Contact.tsx` | All variants: to, search, params, replace |
| `useParams` (nested) | `pages/UserDetail.tsx`, `pages/ProductDetail.tsx` | `useParams({ from })` in child component |
| `Link activeOptions` | `pages/Products.tsx` | `exact + includeSearch` on category tabs |
| `scrollRestoration` + `defaultPendingMs` | `router.tsx` (createRouter) | Same config options as file-based |

#### Gotchas learned
- `getRouteApi('/path')` solves the circular dependency: page files can't import the route object from `router.tsx` (which imports the page) тАФ `getRouteApi()` looks up by path string at runtime, fully typed.
- `createRootRouteWithContext<T>()` uses **double-call** syntax: `createRootRouteWithContext<RouterContext>()({ component: ... })` тАФ the first call sets the type, the second call sets the options.
- `lazyRouteComponent` build output: `LazyPage-xxxx.js` is a separate chunk тАФ confirmed in `npm run build` output.

#### 04_TanStackQuery.tsx тАФ TanStack Query (React Query)

**Demo:** Posts list, single post lookup, parallel user fetch, pagination, add/update/delete, infinite scroll, prefetch

| Concept | Detail |
|---------|--------|
| `QueryClient` + `QueryClientProvider` | Central cache store тАФ created once, wraps the app |

| `useMutation` тАФ update | `setQueryData` patches cache directly тАФ no extra network call |
| `useMutation` тАФ delete | Optimistic update: remove from cache instantly, rollback via `onError` |
| `cancelQueries` | Cancels in-flight refetches before optimistic update to avoid race conditions |
| `useInfiniteQuery` | `data.pages` accumulates pages. `getNextPageParam` returns next page arg |
| `prefetchQuery` | Warms cache on hover тАФ click renders instantly with no spinner |
| `useQueryClient()` | Access `invalidateQueries`, `setQueryData`, `getQueryData`, `prefetchQuery` in components |
| `ReactQueryDevtools` | Floating panel тАФ inspect cache, status, staleTime, refetch live |

**When to use:** Any app fetching from REST/GraphQL APIs тАФ the most popular server-state library in the React ecosystem.

#### 01_MemoOptimization.tsx тАФ Memoization

**Demo:** Live render counters on every component to make re-renders visible

| Concept | Detail |
|---------|--------|
| `React.memo(Component)` | Skip re-render when props unchanged (shallow equal) |
| Custom `areEqual` | `memo(Component, (prev, next) => prev.id === next.id)` тАФ only re-render when id changes |
| `useMemo(fn, deps)` | Cache expensive filter/sort (500-item list with 5ms artificial delay) |
| `useCallback(fn, deps)` | Stable function refs тАФ without it, memo'd children re-render even when nothing changed |
| Common mistakes | Inline `{}` and `[]` literals create new references every render тЖТ break `React.memo` |
| Fix | `useMemo(() => ({ label: "x" }), [])` + `useCallback(() => fn(), [])` |
| When NOT to memo | Simple components, cheap computations, deps change every render тАФ overhead costs more than it saves |

#### 02_CodeSplitting.tsx тАФ Code Splitting & Lazy Loading

**Demo:** 4 interactive demos with simulated network delays

| Concept | Detail |
|---------|--------|
| `React.lazy(() => import('./X'))` | Creates a separate JS chunk, downloads on first render |
| `<Suspense fallback={<Spinner/>}>` | Required wrapper тАФ shows fallback while chunk loads |
| Spinner fallback | Simple animated spinner |
| Skeleton fallback | Shimmer placeholder matching content shape тАФ better for slow loads |

| Suspense boundary placement | One boundary = all wait together. Separate boundaries = each loads independently (better UX) |
